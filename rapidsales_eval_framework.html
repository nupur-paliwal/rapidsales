S-01

Evaluation Framework — Channel Architecture

Each channel has fundamentally different latency, user intent, and measurability constraints. The framework treats them as distinct systems, not as a single AI with three skins.

◎
Design philosophy: Every metric must be computable programmatically without a human reading the output — that's the constraint. Metrics requiring subjective human scoring are labelled as HUMAN-IN-LOOP and used sparingly, only for the quarterly calibration runs, not the live monitoring pipeline.
Email
Personalised Cold Email
Asynchronous · Read at leisure · Can be re-read · Subject line = gatekeeper · 1-way first touch
Subject Line
Open Rate
EM-01
What it measuresWhether the subject line is compelling enough to earn an open in a cold inbox. The single highest-leverage email metric — nothing else matters if it's 0.
Calculation
Opens ÷ Delivered × 100
Tracked via email platform pixel. Segment by: city tier, lead source, time-of-send.
✓ Good
≥ 28%
✗ Poor
≤ 14%
AutomationEmail platform webhook → metrics DB. No human needed.
Personalisation
Accuracy Score
EM-02
What it measuresWhether the AI correctly used the lead's specific data — name, city, monthly bill range, home ownership — without hallucinating details not in the input context.
Calculation
Regex + field-match check:
verified_fields_used ÷ total_personalisable_fields
+ 0 if any hallucinated field detected
✓ Good
≥ 0.95
✗ Poor
≤ 0.80
AutomationPost-generation linter: cross-checks rendered email fields against CRM record. Flags mismatches. Fully automated.
CTA Clarity
Score
EM-03
What it measuresWhether the email contains exactly one clear, specific call-to-action (e.g., "Book a free site visit" link) — not zero, not three competing CTAs, not a vague "let us know".
Calculation
Rule-based classifier:
1 = exactly 1 actionable CTA present
0.5 = CTA present but vague
0 = no CTA or 3+ CTAs
✓ Good
Score = 1.0
✗ Poor
Score ≤ 0.5
AutomationRegex: count links + action verbs near closing. CTA presence classifier using keyword list ("schedule", "book", "reply to this").
Reply-to-Lead
Conversion Rate
EM-04
What it measuresOf all emails that received a reply (any reply — positive, question, objection), how many resulted in a qualified lead being passed to the sales team. The downstream business metric for email quality.
Calculation
Replies_that_became_qualified_leads
÷ Total_email_replies × 100
✓ Good
≥ 18%
✗ Poor
≤ 7%
AutomationCRM pipeline stage tracking. Lead status join on email reply thread ID. Weekly batch computation.
WhatsApp
WhatsApp Conversational AI
Semi-synchronous · Informal tone expected · Multi-turn · Read receipts visible · Short messages norm · High abandon rate
Conversation
Completion Rate
WA-01
What it measuresThe % of initiated WhatsApp conversations that reach a defined terminal state (qualified lead, callback booked, or DNC) rather than going silent mid-thread — the primary quality signal for conversational flow.
Calculation
Conversations_with_terminal_outcome
÷ Total_conversations_started × 100
Terminal state = any of: INTERESTED, CB_BOOKED, DNC flagged, HANDOFF triggered.
✓ Good
≥ 52%
✗ Poor
≤ 28%
Median Response
Latency
WA-02
What it measuresHow fast the AI replies after the user's message timestamp. On WhatsApp, a 4-second delay feels like a dead line — users will leave. This metric has a sweet spot: too fast feels like spam, too slow kills the thread.
Calculation
median(t_ai_reply − t_user_message)
in milliseconds, per conversation cohort
✓ Good
800ms – 2.4s
✗ Poor
<400ms or >5s
AutomationMessage timestamp delta from WhatsApp Business API webhook log. P50 computed hourly.
Turn-Level
Coherence Score
WA-03
What it measuresWhether the AI's reply is topically coherent with the user's preceding message — i.e., it's answering what was actually asked, not continuing a prior tangent or ignoring a mid-thread topic shift.
Calculation
Sentence-BERT cosine similarity
between [user_msg] and [ai_reply]
averaged over all turns in conversation.
Score: 0–1. Batch-computed post-conversation.
✓ Good
≥ 0.68
✗ Poor
≤ 0.45
NoteFully automated via a SBERT embedding pipeline. No human annotation needed at scale.
Objection Recovery
Rate
WA-04
What it measuresWhen the AI detects a soft-objection signal (e.g., "bahut mehnga hai"), what % of those conversations continue past the next 2 turns rather than going silent? Measures objection-handling effectiveness specifically.
Calculation
Conversations_with_≥2_turns_after_objection
÷ Conversations_where_objection_classified × 100
✓ Good
≥ 44%
✗ Poor
≤ 20%
Voice
AI Voice Cold Call
Real-time · No re-read · Tone & pacing matter as much as content · Interruption-sensitive · STT errors compound · 15-second survival window
Post-Opening
Retention Rate
VC-01
What it measuresThe % of calls where the prospect stays on the line past the 15-second mark after the opening — the single most important early signal of script naturalness and trust.
Calculation
Calls_with_call_duration > 15s
÷ Calls_answered × 100
Calls_answered = pickup with >2s of audio (not IVR/instant-hangup).
✓ Good
≥ 58%
✗ Poor
≤ 30%
AutomationCall duration from telephony API (Twilio/Exotel). Computed per call batch, hourly rollup.
Barge-In
Recovery Rate
VC-02
What it measuresWhen the prospect interrupts the AI mid-sentence (barge-in detected), what % of those instances does the AI successfully stop, acknowledge, and continue the conversation vs. talking over the prospect or going silent?
Calculation
Barge-ins_followed_by_≥1_prospect_turn
÷ Total_barge-in_events × 100
Barge-in = VAD overlap event logged by telephony layer.
✓ Good
≥ 72%
✗ Poor
≤ 45%
Intent Classification
Accuracy
VC-03
What it measuresHow accurately the NLU layer classifies the prospect's intent against the 6 defined outcomes (INTERESTED, CALLBACK, SOFT_OBJ, HARD_DNC, WRONG_NUMBER, NO_ANSWER). Measured on labelled test set + sampled live calls.
Calculation
Correct_classifications ÷ Total_classified
Weighted by class (DNC errors penalised 3×
due to compliance risk)
✓ Good
≥ 91%
✗ Poor
≤ 80%
NoteWeekly 1% sample reviewed by QA agent to maintain ground-truth label quality. Not fully automated — HUMAN-IN-LOOP for label refresh only.
Warm Handoff
Drop Rate
VC-04
What it measuresOf calls where the AI initiated a warm handoff to a human rep, what % resulted in the prospect hanging up before the rep completed their first full sentence? The direct quality signal for handoff script and timing.
Calculation
Handoffs_where_call_ends_within_15s_of_rep_intro
÷ Total_handoffs_initiated × 100
✓ Good
≤ 8%
✗ Poor
≥ 22%
AutomationCall segment timestamps from telephony. Rep-joined event + call_end event delta.
Cross-Channel
Cross-Channel Business Metrics
Channel-agnostic · Business outcome layer · Lagging indicators that validate leading metrics
Qualified Lead
Rate (QLR)
CC-01
What it measuresAcross all channels, the % of outreach attempts that produce a qualified lead passed to the sales team. The primary business health metric — if this drops, something upstream is breaking.
Calculation
Qualified_leads_generated
÷ Total_outreach_attempts × 100
Computed per channel + combined.
✓ Good
≥ 6.5%
✗ Poor
≤ 2.5%
Brand Safety
Violation Rate
CC-02
What it measuresThe % of AI outputs that contain a brand-safety violation: competitor mention, price claim not in approved range, legally sensitive warranty statement, hallucinated product detail. Zero tolerance metric.
Calculation
Outputs_with_1+_violation_flag
÷ Total_AI_outputs × 100
Detected by rule-based scanner + LLM judge.
✓ Good
≤ 0.1%
✗ Poor
≥ 0.5%
NoteAny single week at ≥ 0.5% triggers a P1 freeze on all outreach until root cause is identified.
S-03

Synthetic Test Dataset — 3 Stress Cases

These are not happy-path tests. Each case is designed to expose a specific AI failure mode. Easy inputs are not useful for evaluation — they tell you nothing about where the system breaks.

⚠
Design principle: A test set composed entirely of "nice prospect asks a clear question" scenarios gives you false confidence. Real cold outreach has: people who are annoyed, people who ask adversarial questions, ambiguous signals, and multi-intent messages. The three cases below each stress a different failure vector.
Email

TC-01 · Sparse Lead Data — Personalisation Hallucination Trap

Failure Mode: Hallucination
Input Context (CRM record passed to AI)
name: "Ramesh K."
city: "Nagpur"
monthly_bill: NULL
home_type: NULL
lead_source: "Facebook form — partial fill"
ownership: NULL
previous_contact: None
Test input — what the AI is asked to generate
Generate a cold personalised email for this lead to introduce SunPower Solar and get a reply.
Ideal AI Response (what we score against)
Subject: Nagpur homeowners are saving ₹2,000/month on electricity — here's how

Hi Ramesh,

I'm reaching out from SunPower Solar. We've been helping homeowners in Nagpur reduce their electricity bills significantly — many are saving between ₹1,800 and ₹3,500 every month after switching to rooftop solar, thanks in part to the government's 40% subsidy under PM Surya Ghar.

If you own your home and your monthly electricity bill is ₹2,000 or above, there's a good chance solar makes strong financial sense for your situation.

Would it be okay if I sent you a quick personalised estimate? Just reply with your approximate monthly bill and I'll have something ready for you in a few hours.

— Arjun, SunPower Solar
Scoring dimensions
EM-02 Personalisation: No invented fields EM-03 CTA: "Reply with your bill" — exactly 1 CC-02 Safety: No false ownership assumption Tone: City localised, not generic
Why this is hard: Most AI systems will hallucinate a monthly bill (e.g., "I see your bill is around ₹3,000") to make the email feel more personalised. The ideal response turns the NULL fields into a qualification question — which is more honest AND more likely to get a reply. Hallucinating any field here scores 0 on EM-02 regardless of how good the rest of the email is.
WhatsApp

TC-02 · Double-Objection Mid-Thread — Coherence + Recovery Test

Failure Mode: Context Loss
Prior conversation context (5-turn thread)
Turn 1 AI: Opening message + bill question
Turn 2 User: "Haan, bill 3500 aa raha hai hamare ghar ka"
Turn 3 AI: Acknowledged, explained subsidy + EMI
Turn 4 User: "ok interesting... lekin 5 saal pehle padosi ne lagwaya tha, kuch khaas fayda nahi hua"

[Two objections in one message: cost scepticism + social proof failure]
Test input — the exact user message to respond to
"Ok interesting... lekin 5 saal pehle padosi ne lagwaya tha, kuch khaas fayda nahi hua unhe. Aur mujhe lagta hai humara bill toh waise bhi zyada nahi hai 3500 mein."
Ideal AI Response
That's a fair point — and honestly, 5 years ago the economics were quite different. Panel efficiency has improved by around 30–40% since then, and more importantly, the PM Surya Ghar subsidy didn't exist at that time — so your neighbour would have paid full price with older, slower technology.

On your bill — ₹3,500 a month is actually ₹42,000 a year. Over 5 years that's over ₹2 lakh just in electricity. A rooftop system, after subsidy, would cost roughly the same — but then give you 20 more years of near-zero bills.

I'm not asking you to decide anything now — would it be okay if I just shared a rough calculation specific to your home? 2 minutes, no obligation.
Scoring dimensions
WA-03 Coherence: Addresses both objections WA-04 Recovery: Moves toward CTA despite dual objection Accuracy: Does not dismiss neighbour's experience Tone: Short, WhatsApp-native length
Why this is hard: Two objections arrive simultaneously — social proof failure ("my neighbour tried it") AND price minimisation ("3500 isn't much"). Most AI systems pick ONE to respond to and ignore the other, which feels dismissive. The ideal response reframes both AND converts the sceptical data point (neighbour's experience) into supporting evidence by using the technology-age argument. A response that only addresses one objection scores 0.5 on WA-03.
Voice

TC-03 · Suspicious Prospect — DNC Boundary + Trust Recovery

Failure Mode: DNC Misclassification
Call context
call_attempt: 1st
time: 11:30 AM weekday
lead_tier: Tier 2 — Indore
prior_contact: None
AI_opening_completed: Yes — 12 seconds
prospect_tone: [detected: guarded/flat]
Test input — exact spoken words (post-opening)
"Aapko mera number kahan se mila? Mera naam kisne diya? Main nahi chahta ki mujhe aisi calls aayein. Ye sab scam hota hai."
Ideal AI Response (spoken)
[0.4s pause before responding — acknowledges weight of question]

"Aapka sawaal bilkul sahi hai — aur main samajhta hoon ye frustrating hota hai. [0.3s] Aapka number ek registered homeowner database se hai jo TRAI guidelines ke under kaam karta hai — same way banks aur insurance companies contact karte hain. Hum kabhi bhi numbers share ya sell nahi karte.

Aur agar aap chahte hain ki aapko dobara kabhi call na aaye — main abhi aapka number block kar sakta hoon, koi sawaal nahi. Bas ek word boliye.

Ya agar aap sunna chahein ki baat kya thi — 30 second mein bata sakta hoon."
Scoring dimensions + critical classification check
VC-03 Intent: Must NOT classify as DNC yet CC-02 Safety: No made-up data source details Tone: Pause before reply — not instant Structure: Offers DNC + continues — not forced
Why this is hard: "Main nahi chahta ki mujhe aisi calls aayein" sounds like a DNC, but the prospect ends with "Ye sab scam hota hai" — which is an objection seeking reassurance, not a definitive opt-out. Misclassifying this as DNC is a false negative (lost lead). Ignoring the DNC signal and continuing the pitch is a compliance failure. The ideal response acknowledges both possibilities and hands the choice back to the prospect — it scores on VC-03 only if the classification is SOFT_OBJECTION (not DNC) and the response is appropriately bifurcated.
S-04

Quality Drift Detection System — Automated

Designed to run with zero human review for daily monitoring. Human-in-loop only for P1 incidents and weekly calibration. Built for a 3-person ops team, not a 30-person quality org.

L1

Real-Time Rule Engine

What it checks (every output): Brand safety violations (regex), hallucinated field values (field-match linter), missing CTA in email, response length anomaly (>3× baseline), DNC keyword in non-DNC classified call.

Latency: <100ms. Fires before output is delivered if pre-send check is implemented; otherwise fires in background and flags for review queue.

Output: Pass / Fail per rule. Fail rate tracked hourly.
L2

Statistical Drift Monitor

What it checks (hourly batch): All 12 metrics computed on rolling 7-day window vs. 30-day baseline. Any metric with a Z-score >2.5 against its baseline triggers a DRIFT_ALERT.

Special case: QLR is computed weekly (insufficient daily volume). All other metrics hourly.

Output: Per-metric drift score. Heatmap written to ops dashboard. Slack alert if any drift alert fires.
L3

LLM-as-Judge Sampling

What it checks (daily 2% sample): A judge LLM (GPT-4o or Claude-3.5) scores sampled outputs on: tone appropriateness, coherence, objection quality, and brand alignment. Uses a fixed rubric and outputs a numeric score 0–10.

Threshold: If daily mean judge score drops >0.8 points vs. 14-day rolling mean → JUDGE_DRIFT alert.

Cost: ~₹400/day at 2% sample rate of 5,000 daily outputs.
◎
Why LLM-as-Judge for sampling? Human annotation at 2% of 5,000 daily outputs = 100 reviews/day = 1 FTE. LLM judge does the same in 8 minutes for ₹400. The tradeoff: the judge LLM itself can drift if not calibrated — so monthly calibration against 50 human-labelled "gold standard" conversations is mandatory to catch judge model degradation.
Alert Trigger Definitions
Metric / Signal Trigger Condition Sev Channel Automated Action
CC-02 Brand Safety Weekly violation rate ≥ 0.5% OR any single output with legal/warranty hallucination P1 All Pause all outreach. Slack #incidents. PagerDuty to on-call engineer. Block output delivery if pre-send check enabled.
VC-03 Intent Accuracy Weekly accuracy drops >5 percentage points from 30-day baseline P1 Voice Pause voice outreach. Force 100% human review of intent classifications until root cause found. CRM: flag all recent classifications for audit.
EM-01 Open Rate 7-day rolling open rate drops >6 percentage points from 30-day baseline (Z > 2.5) P2 Email Slack alert to email team. A/B test new subject line variants. Do not pause — but log and monitor daily.
WA-01 Completion Rate 7-day completion rate drops below 28% or Z > 2.5 from baseline P2 WhatsApp Slack alert. Pull sample of abandoned conversations. Check if a specific turn is causing drop-off (turn-level funnel analysis). Review WA-03 coherence scores same period.
VC-01 Retention Rate Post-opening retention drops below 30% on any 6-hour cohort P2 Voice Flag cohort for review. Check if TTS voice quality degraded (telephony provider issue vs. script issue). Rollback opening script to last stable version if voice provider confirmed OK.
L3 LLM Judge Score Daily mean judge score drops >0.8 from 14-day rolling mean P2 All Slack alert. Pull 20 lowest-scored outputs for human review. Identify if degradation is concentrated in one channel, one scenario type, or system-wide. Escalate to P1 if confirmed system-wide.
EM-02 Personalisation Accuracy Daily hallucination rate (any invented field) exceeds 2% P3 Email Slack alert to AI team. Review field-linter logs for which fields are being hallucinated. Patch prompt. Monitor for 48h before clearing alert.
WA-02 Response Latency Median latency >5s for any 1-hour window P3 WhatsApp Infrastructure alert — likely LLM provider latency spike. Switch to fallback model or cached response flow. Alert infra team.
P1 Incident Response Protocol
Live Alert — P1 Response Runbook
T+0
Automated: Outreach paused on affected channel(s). PagerDuty fires to on-call engineer. Slack #incidents message with: metric triggered, current value, baseline, timestamp, sample output links.
T+15m
On-call engineer: Confirms alert is genuine (not a data pipeline error). Pulls 10 most recent outputs from affected channel. Classifies root cause: prompt regression, model provider change, data pipeline issue, or CRM data corruption.
T+30m
Decision gate: If root cause = prompt regression → rollback to last known-good prompt version (git-tagged). If root cause = model provider → switch to fallback model. If root cause = data pipeline → freeze outreach until data issue resolved. If unknown → keep paused, escalate to AI lead.
T+2h
Validation run: Run synthetic test suite (TC-01, TC-02, TC-03 + 20 additional regression cases) against the fixed version. If all pass → resume outreach with 5% of normal volume. Monitor L1 + L2 metrics for 1 hour before full resume.
T+24h
Post-mortem: 1-page incident report: timeline, root cause, detection lag (how long did the model degrade before the alert fired?), fix applied, regression test coverage gap identified. Any test coverage gap → add new test case to the synthetic suite before closing ticket.
Implementation Stack — Day 1 Lean Team
L1 Rule Engine
Python linter + regex pipeline
Run as a Lambda/Cloud Function on each output. Git-managed rule definitions so any team member can add a rule via PR. Results written to Postgres. Dashboard: Metabase or Grafana on top. No specialised ML infra needed — Day 1 ready.
L2 Stat Monitor
dbt + hourly cron + Z-score script
dbt models materialise the 12 metrics from raw event logs. A Python cron script (runs hourly) computes Z-scores vs. 30-day windows and writes alerts to a Slack webhook. Entire system is <200 lines of Python. One engineer can build in 3 days.
L3 LLM Judge
Batch job — Anthropic/OpenAI API
Daily 2% sample selected randomly per channel. Structured prompt with fixed rubric (0–10 per dimension). Scores written to eval DB. Monthly calibration: 50 gold-standard conversations hand-labelled by QA lead used to validate judge consistency (Spearman ρ > 0.8 required).
Prompt Versioning
Git tags + deployment registry
Every prompt deployed to production has a semver tag (e.g., v2.3.1-email-cold). Rollback = one command. Each tag links to: eval scores at deployment, test suite results, deploying engineer. This is the most important operational hygiene item — without it, rollback takes hours not minutes.
